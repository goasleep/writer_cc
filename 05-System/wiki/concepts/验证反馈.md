# 验证反馈

**英文名**: Validation & Feedback
**类型**: Agent Harness组件
**比喻**: Agent的纠错机制

## 概述

验证反馈是Agent Harness中提供外部反馈回路的机制。根据[Agent Harness 综述：同一个模型，为什么做做出的 Agent 差这么远](../sources/Agent Harness 综述：同一个模型，为什么做做出的 Agent 差这么远.md)的论述，工具给了模型行动能力，验证才给了模型纠错能力。

## 核心观点

### Boris Cherny的洞察
> 给模型一种能验证自己工作的方式，质量提升2到3倍。

### 行动能力 vs 纠错能力
- **工具**: 给模型行动能力
- **验证**: 给模型纠错能力

### 为什么需要外部验证
**问题**: 让生成器自己验自己，速度很快，但最容易放过自己

**解决**: 只要任务碰到代码、页面、部署、数据写入这些"能真测"的地方，外部验证几乎都是值得补上的。

## 验证类型

### 1. 代码验证
```python
# 单元测试
def run_unit_tests(code):
    return pytest.main(["-q", "tests/"])

# 类型检查
def type_check(code):
    return mypy.run(code)

# Lint检查
def lint_code(code):
    return pylint.run(code)
```

### 2. 功能验证
```python
# 端到端测试
def run_e2e_test(scenario):
    return playwright_test(scenario)

# API测试
def test_api(endpoint):
    return requests.get(endpoint).status_code == 200
```

### 3. UI验证
```python
# 页面截图
def take_screenshot(url):
    return browser.screenshot(url)

# 视觉回归测试
def visual_regression(before, after):
    return compare_images(before, after)
```

### 4. 部署验证
```python
# 部署检查
def check_deployment(service):
    return k8s.is_ready(service)

# 健康检查
def health_check(endpoint):
    return requests.get(f"{endpoint}/health").ok
```

## 反馈回路

### 1. 即时反馈
```python
# 工具执行后立即验证
result = execute_tool("deploy", params)
validation = validate_deployment(result)

if not validation["success"]:
    # 提供错误信息给模型
    return {
        "status": "failed",
        "error": validation["error"],
        "suggestion": "检查部署配置"
    }
```

### 2. 延迟反馈
```python
# 批量验证
results = []
for action in actions:
    results.append(execute(action))

# 统一验证
validation_summary = validate_all(results)
return validation_summary
```

### 3. 异步反馈
```python
# 后台验证任务
async def validate_async(work):
    job_id = queue_validation_job(work)
    return job_id

# 查询验证结果
def get_validation_result(job_id):
    return check_job_status(job_id)
```

## 验证策略

### 1. 自验证 vs 外部验证

| 维度 | 自验证 | 外部验证 |
|------|--------|----------|
| 速度 | 快 | 慢 |
| 客观性 | 低（容易放过自己） | 高 |
| 适用场景 | 简单任务 | 复杂任务 |
| 代码相关 | 不推荐 | **推荐** |

### 2. 持续验证
```python
# 每一步都验证
for step in plan:
    result = execute(step)
    validation = validate(result)
    if not validation.passed:
        handle_failure(validation)
```

### 3. 阶段性验证
```python
# 关键节点验证
major_milestones = ["implementation", "testing", "deployment"]
for milestone in major_milestones:
    complete_phase(milestone)
    validate_milestone(milestone)
```

## 实现模式

### 1. 测试驱动
```python
# 先写测试
test = generate_test(requirement)

# 再实现
implementation = generate_code(test)

# 运行测试验证
result = run_test(test, implementation)
return result
```

### 2. 对比验证
```python
# 修改前后对比
before = read_file("file.py")
after = apply_change("file.py", change)

diff = compare(before, after)
validation = validate_diff(diff)
```

### 3. 快照验证
```python
# 保存期望状态
expected_state = capture_state()

# 执行操作
execute_action(action)

# 验证当前状态
actual_state = capture_state()
validation = validate_states(expected_state, actual_state)
```

## 最佳实践

### 1. 能写测试就别只写总结
```python
# 差的做法
return "代码应该没问题"

# 好的做法
test_result = run_tests(code)
return {
    "tests_run": test_result.num_tests,
    "tests_passed": test_result.passed,
    "tests_failed": test_result.failed,
    "coverage": test_result.coverage
}
```

### 2. 能跑页面就别只看文字
```python
# 差的做法
return "页面应该正常显示了"

# 好的做法
screenshot = take_screenshot(url)
visual_test = compare_screenshot(screenshot, expected)
return visual_test
```

### 3. 让客观证据说话
```python
# 不让模型自己说"应该没问题"
# 而是让lint、类型检查、截图或真实API响应说话
validation_results = {
    "type_check": mypy.run(code),
    "lint": pylint.run(code),
    "screenshot": browser.screenshot(url),
    "api_test": api.post(endpoint).json()
}
return validation_results
```

## 常见陷阱

### 1. 过度依赖模型自评
- 模型容易放过自己
- 缺少客观标准
- 难以发现边界情况

### 2. 验证太晚
- 所有操作完成才验证
- 失败代价高
- 难以定位问题

### 3. 验证覆盖不足
- 只验证主路径
- 忽略边界情况
- 缺少异常测试
