# 缺页异常

**英文名**: Page Fault
**类型**: 硬件异常
**触发者**: CPU（内存管理单元MMU）
**处理者**: 操作系统内核

## 概述

缺页异常是当程序访问的虚拟内存页不在物理内存中，或者访问权限不满足时，CPU触发的一种硬件异常。根据[不懂 fork 与 写时复制，别再说你懂 Linux 内核了](../sources/不懂 fork 与 写时复制，别再说你懂 Linux 内核了.md)的论述，在写时复制机制中，缺页异常是触发内存复制的关键信号。

## 触发原因

### 1. 页面不在物理内存（按需分页）
- 虚拟页未加载到物理内存
- 页面被换出（swap out）
- 首次访问（延迟加载）

### 2. 访问权限不匹配
- **写时复制**: 尝试写入只读页
- **保护违规**: 用户态访问内核态内存
- **权限不足**: 读取不可读页

## 在COW中的作用

### 触发流程
```
1. fork完成，父子进程共享物理页
   ↓
2. 共享页被标记为只读
   ↓
3. 子进程尝试写操作: arr[4] = 20;
   ↓
4. CPU检测到写操作违反只读权限
   ↓
5. 触发缺页异常（Page Fault）
   ↓
6. 内核捕获异常，判断是COW相关
   ↓
7. 分配新物理页，复制数据，更新页表
   ↓
8. 返回用户态，重新执行写操作
   ↓
9. 写操作成功完成
```

### 关键代码示意
```c
// 用户态代码
arr[4] = 20;  // 尝试写只读页
  ↓
// CPU触发异常，进入内核
内核异常处理程序:
  if (是COW页) {
    new_page = allocate_physical_page();
    copy_data(old_page, new_page);
    update_page_table(virtual_addr, new_page, WRITABLE);
    return;  // 返回用户态重新执行指令
  }
```

## 缺页异常处理流程

### 1. 异常触发
- CPU执行指令时检测到异常条件
- 保存当前上下文（寄存器、程序计数器等）
- 切换到内核态

### 2. 内核处理
```c
void page_fault_handler(void *fault_address) {
    // 1. 检查故障地址是否合法
    if (!is_valid_address(fault_address)) {
        SIGSEGV();  // 段错误
        return;
    }
    
    // 2. 检查是否是COW页
    if (is_cow_page(fault_address)) {
        handle_cow_fault(fault_address);
        return;
    }
    
    // 3. 按需分页：从磁盘加载页
    if (!is_present(fault_address)) {
        load_page_from_disk(fault_address);
        return;
    }
    
    // 4. 权限错误
    SIGSEGV();
}
```

### 3. 恢复执行
- 内核修正问题（分配页、更新权限等）
- 恢复用户态上下文
- 重新执行触发异常的指令

## COW缺页异常的特殊处理

### 判断依据
内核如何判断缺页异常是由COW引起的？

1. **页表项检查**: 页表项标记为COW
2. **权限检查**: 页是只读的，但进程有写入权限（逻辑上）
3. **引用计数**: 物理页被多个进程共享

### 处理步骤
```c
void handle_cow_fault(void *virtual_addr) {
    // 1. 获取旧的物理页
    void *old_phys_page = get_physical_page(virtual_addr);
    
    // 2. 分配新的物理页
    void *new_phys_page = allocate_physical_page();
    
    // 3. 复制页内容
    memcpy(new_phys_page, old_phys_page, PAGE_SIZE);
    
    // 4. 更新当前进程的页表
    update_page_table_entry(
        virtual_addr,      // 虚拟地址
        new_phys_page,     // 新物理地址
        PAGE_READWRITE     // 可写权限
    );
    
    // 5. 减少旧页的引用计数
    decrement_ref_count(old_phys_page);
}
```

## 性能影响

### 单次缺页开销
- **上下文切换**: 用户态↔内核态
- **内存分配**: 分配新物理页
- **数据复制**: 4KB数据复制（典型页大小）
- **页表更新**: 更新页表项和TLB

**典型耗时**: 数百纳秒到数微秒

### 优化策略
1. **批量处理**: 处理连续的缺页异常
2. **预读**: 预加载可能访问的页
3. ** hugepage**: 使用大页减少页表开销
4. **TLB优化**: 减少TLB miss

## 与其他异常的区别

### vs 段错误（Segmentation Fault）
| 特性 | 缺页异常 | 段错误 |
|------|----------|--------|
| 触发条件 | 页不在内存或权限问题 | 访问非法地址 |
| 可恢复 | 是（处理后可继续） | 否（进程终止） |
| 信号 | SIGSEGV（但可处理） | SIGSEGV（不可恢复） |
| 常见场景 | COW、按需分页 | 空指针、数组越界 |

### vs 一般保护异常（General Protection Fault）
| 异常类型 | 缺页异常 | 一般保护异常 |
|---------|----------|-------------|
| 触发者 | MMU | CPU |
| 原因 | 内存页问题 | 权限、模式、段等问题 |
| 编号 | 14 | 13 |
| 处理 | 页错误处理程序 | 通用保护处理程序 |

## 调试与监控

### 查看缺页异常统计
```bash
# Linux
ps -o min_flt,maj_flt <pid>  # 次要缺页和主要缺页
/proc/<pid>/stat               # 详细统计
perf record -e page-faults     # 性能分析
```

### 缺页类型
1. **次要缺页（Minor Fault）**:
   - 页在物理内存中（如COW页）
   - 只需页表操作
   - 开销较小

2. **主要缺页（Major Fault）**:
   - 页不在物理内存（在swap/磁盘）
   - 需要磁盘I/O
   - 开销较大

## 常见问题

### 1. 缺页异常频繁
**原因**:
- 内存不足，频繁换页
- COW页被频繁修改
- 访问模式不佳（随机访问）

**解决**:
- 增加物理内存
- 优化数据结构
- 改善访问局部性

### 2. 缺页异常导致性能下降
**诊断**:
```bash
perf top -e page-faults
perf record -e page-faults -g
```

**优化**:
- 减少COW页的修改
- 使用mlock锁定关键页
- 调整工作集大小
