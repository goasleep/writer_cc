# In-Memory Index

**中文**: 内存索引
**复杂度**: O(1)（哈希表）
**类型**: 数据结构

## 概述

内存索引是将数据完全加载到RAM中，使用哈希表等数据结构提供O(1)查找性能的存储策略。根据[Do You Even Need a Database?](../sources/Do You Even Need a Database.md)的测试，它是性能最高的存储方案，但受限于可用内存大小。

## 工作原理

### 实现方式
1. **启动时加载**: 读取整个数据文件一次
2. **哈希表存储**: 将每条记录存储在以ID为key的哈希表中
3. **双重写入**: 写入时同时更新map和文件
4. **持久化**: 文件是持久化备份，map是索引
5. **重启恢复**: 进程重启后从文件重新加载

### 并发控制
- **Go**: `sync.RWMutex` - 允许多个读者并行
- **Rust**: `RwLock` - 读写锁保护
- **Bun**: JavaScript内置并发控制

## 性能特征

### 基准测试结果（Go实现）
- **10k记录**: 97,040 req/s
- **100k记录**: 98,277 req/s
- **1M记录**: 97,829 req/s
- **延迟**: 亚毫秒级（< 1ms）
- **一致性**: 性能在任何规模下保持稳定

### 性能天花板
> "If your dataset fits in RAM, nothing on disk will match it."

内存哈希表是性能的上限，任何磁盘方案都无法匹敌。

## 优势

### 1. 极致性能
- O(1)查找复杂度
- 亚毫秒延迟
- 数据规模增长不影响性能

### 2. 实现简单
- 标准库哈希表
- 无需复杂索引结构
- 代码易于理解和维护

### 3. 并发友好
- 读写锁允许多读者并行
- 无磁盘I/O竞争
- 低锁争用

### 4. 实时性
- 无磁盘寻址延迟
- 无缓存未命中惩罚
- 确定性性能

## 限制

### 1. 内存限制
**启动时必须加载全部数据**：
- 几百万条小记录没问题
- 几千万条需要几GB RAM
- 受限于物理内存大小

### 2. 启动时间
数据集越大，启动时间越长：
- 需要解析整个文件
- 构建哈希表
- 可能影响冷启动性能

### 3. 单进程限制
- 每个进程有自己的map副本
- 多实例会数据分歧
- 需要外部数据源协调

### 4. 内存成本
- 数据增长需要更多RAM
- 可能需要升级服务器
- 内存成本比磁盘高

## 适用场景

### 适合使用的情况
1. **数据集适合RAM**: 几百万条记录或几GB数据
2. **读取密集**: 读多写少的场景
3. **单实例部署**: 不需要多进程协调
4. **快速启动**: 启动时间可接受
5. **预算充足**: 内存成本可接受

### 不适合的情况
1. **数据集过大**: 超过可用RAM
2. **多实例部署**: 需要多进程数据共享
3. **内存受限**: 嵌入式系统或低成本服务器
4. **频繁重启**: 需要快速启动时间

## 语言和运行时差异

### Go vs Bun vs Rust
根据[Do You Even Need a Database?](../sources/Do You Even Need a Database.md)的测试：

| 实现语言 | 性能 (req/s) | 说明 |
|---------|-------------|------|
| Go | 97,040 | 标准库sync.RWMutex |
| Bun (JavaScript) | 106,000 | JavaScriptCore + Zig HTTP层 |
| Rust | 169,000 | 最低I/O开销 + 快速JSON反序列化 |

**关键洞察**:
- Bun击败Go: 运行时优化（JavaScriptCore + Zig）
- Rust大幅领先: serde JSON反序列化 + 低I/O开销
- **语言不如运行时重要**

## 容量规划

### 内存使用估算
假设每条记录500字节：
- 10万条: ~50MB
- 100万条: ~500MB
- 1000万条: ~5GB
- 1亿条: ~50GB

### 性能容量
根据[Do You Even Need a Database?](../sources/Do You Even Need a Database.md)的推算：
- **97k req/s**: 可支撑**3.49亿日活用户**
- **假设**: 10%日活同时在线，每小时10次查询

## 迁移策略

### 从内存索引迁移到数据库
当数据集超过RAM时：
1. **增量迁移**: 将旧数据导入数据库
2. **双写期**: 同时写入内存和数据库
3. **逐步切换**: 将读取流量切换到数据库
4. **完全迁移**: 关闭内存索引

### 最佳实践
- **提前规划**: 在达到内存限制前规划迁移
- **监控增长**: 追踪数据增长速度
- **预留时间**: 迁移需要时间测试和验证
