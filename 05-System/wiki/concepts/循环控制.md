# 循环控制

**英文名**: Loop Control
**类型**: Agent Harness组件
**所属层**: Harness六大承重层之一

## 概述

循环控制是Agent Harness的核心心脏层。根据[Agent Harness 综述：同一个模型，为什么做出来的 Agent 差这么远](../sources/Agent Harness 综述：同一个模型，为什么做出来的 Agent 差这么远.md)的论述，虽然表面上只是一个while loop，但难点在于循环里每一步到底由谁控制、何时终止、出错后怎么回来。

## 基本结构

### 表面实现
```
while (任务未完成):
    1. 组装输入
    2. 调用模型
    3. 解析输出
    4. 执行工具
    5. 把结果塞回去
```

### 实际难点

#### 1. 控制权问题
- 每一步由谁控制？
- 模型、系统、用户还是外部触发器？

#### 2. 终止条件
- 何时终止？
- 成功完成？失败退出？超时？资源耗尽？

#### 3. 错误恢复
- 出错后怎么回来？
- 重试？回滚？求助用户？跳过步骤？

## 常见问题

### 1. 无限转圈
**原因**: 终止条件不清晰或无法满足

**示例**: 
- 模型不断调用同一个工具但总是失败
- 任务目标与当前状态不匹配

### 2. 提前收尾
**原因**: 模型误解了中间结果为最终结果

**示例**:
- 只完成了部分步骤就认为任务完成
- 没有验证就认为目标已达成

### 3. 中间结果当最终结果
**原因**: 缺少验证机制

**示例**:
- 输出了计划但没有执行
- 生成了代码但没有测试

## 设计要点

### 明确的终止条件
```python
# 好的设计
termination_conditions = [
    "目标达成",
    "达到最大迭代次数",
    "用户取消",
    "遇到无法恢复的错误",
    "超时"
]
```

### 清晰的状态机
```python
# 状态跟踪
states = ["idle", "planning", "executing", "verifying", "completed", "failed"]
current_state = "idle"
state_transitions = {
    "idle": ["planning"],
    "planning": ["executing", "failed"],
    "executing": ["verifying", "planning", "failed"],
    "verifying": ["completed", "executing", "failed"]
}
```

### 错误恢复策略
1. **可重试的错误**: 自动重试（带次数限制）
2. **可恢复的错误**: 回退到上一个状态
3. **不可恢复的错误**: 终止并报告
4. **需要人工介入**: 暂停等待用户输入

## 实现模式

### ReAct循环
```
思考 → 行动 → 观察 → 思考 → 行动 → 观察...
```
- 边想边做
- 灵活但可能不稳定

### Plan-and-Execute循环
```
规划 → 执行 → 验证 → 下一步规划...
```
- 先画施工图再开工
- 稳定但缺乏灵活性

### 混合模式
结合两者优点：
1. 初始规划（Plan）
2. 执行并观察（Execute）
3. 偏差时重新规划（ReAct）
4. 继续执行（Execute）

## 性能考量

### 循环次数优化
- 避免无效循环（重复相同的失败操作）
- 设置合理的最大迭代次数
- 检测循环模式并提前终止

### 状态检查开销
- 最小化每次循环的状态检查
- 异步化耗时操作
- 缓存不变的状态信息
