# Agent Harness 综述：同一个模型，为什么做出来的 Agent 差这么远

## 核心论点

同一个模型、同一组权重，只换外面的系统层（Harness），Agent表现就可能大幅跳跃。Harness是在模型和真实交付之间补上的可运行、可恢复、可验证、可治理的软件系统。生产级Agent的差距，往往来自模型外面的系统有没有把任务变得可见、动作变得可控、状态变得可恢复、结果变得可验证。

## 核心定义

### Harness
> Harness是在模型和真实交付之间，补上一套可运行、可恢复、可验证、可治理的软件系统。

**工程问题**: 怎样把一个无状态、会推理的模型，变成一个能持续交付结果的系统。

**类比**: Beren Millidge 2023年的类比 - 裸的LLM是一颗没有RAM、没有磁盘、没有I/O的CPU。上下文窗口充当内存，外部数据库充当磁盘，工具集充当设备驱动。

### 三层关系
- **Prompt**: 更像指令
- **Context**: 更像喂给模型的工作台
- **Harness**: 更像操作系统

## Harness六大承重层

### 1. 循环控制（Loop Control）
**表面**: while loop - 组装输入，调用模型，解析输出，执行工具，把结果塞回去，再来一轮

**难点**: 
- 每一步由谁控制
- 何时终止
- 出错后怎么回来

**常见问题**: 无限转圈、提前收尾、误把中间结果当最终结果

### 2. 工具系统（Tool System）
**不只是"给几个函数名"**

需要管理四件事：
1. 工具如何注册
2. 参数如何校验
3. 执行环境是否隔离
4. 结果如何回写成模型能理解的Observation

**分类对照**:
- **Claude Code**: 文件操作、搜索、执行、Web访问、代码智能、子Agent派生（六类）
- **OpenAI Agents SDK**: 函数工具、托管工具、MCP工具

**关键**: 模型知道能调用什么工具只是起点。能否在合适时候调用、带着正确参数调用、失败后继续恢复，才是Harness的事。

### 3. 记忆管理（Memory Management）
**该记什么、什么时候记、什么时候删**

**短期记忆**: 当前会话历史
**长期记忆**: 跨会话持久化的事实、决策和索引

**Claude Code三层做法**:
1. 始终加载的轻量索引（每条约150字符）
2. 按需拉取的详细主题文件
3. 只通过搜索访问的原始会话记录

**核心原则**: 
> 成熟系统不会把记忆当真相，而是把它当线索。先靠记忆提示方向，再回到真实文件、真实环境和真实状态里确认。

**警告**: 记忆如果不能被验证，就很容易从"帮助检索"滑向"帮助幻觉"。

### 4. 状态管理（State Management）
**任务一旦变长，状态管理就从"有最好"变成"没有不行"**

系统需要知道：
- 当前做到哪一步
- 失败后从哪恢复
- 哪些中间产物值得保留
- 哪些只是临时噪音

**实现方式**:
- 结构化状态对象做checkpoint
- git commit、进度文件、任务日志一起当作恢复点

**目的**: 让长任务可以继续，而不是每次都从头赌一遍。

### 5. 权限控制（Permission Control）
**一个看起来很能干的Agent，如果没有权限控制，本质上只是一个事故放大器**

**关键拆分**:
- **模型负责**: 提出动作
- **工具系统负责**: 决定动作能不能做、要不要用户确认、失败后如何处理

**生产环境**: 高风险动作必须有边界

### 6. 验证反馈（Validation & Feedback）
**工具给了模型行动能力。验证才给了模型纠错能力。**

**外部反馈回路**: 测试、lint、类型检查、页面截图、端到端操作、专门的评估器

**Boris Cherny观点**: 给模型一种能验证自己工作的方式，质量提升2到3倍。

**警告**: 没有验证，Harness很容易变成"更快地产出错结果"。

## 完整Harness循环（7步）

1. **接收任务**: 用户输入
2. **组装上下文**: 拉取相关记忆、代码、环境状态
3. **调用模型**: 组装输入，调用LLM
4. **解析输出**: 提取动作、工具调用、思考过程
5. **执行动作**: 工具调用，带权限校验
6. **验证结果**: 测试、lint、截图等外部验证
7. **更新状态**: 更新记忆、写入checkpoint、判断是否终止

**关键**: 循环本身往往很朴素。复杂的是每一步背后的工程取舍。

## 为什么Harness现在被重视

### 两个信号

**信号1**: 同一模型、同一组权重，只换外围系统层，表现大幅跳跃
- LangChain只换外围基础设施，TerminalBench 2.0从前30名外拉到第5
- 研究项目把"优化Harness"当搜索对象，拿到76.4%通过率

**信号2**: 长任务的误差快速累积
- 10步流程，每步99%成功率，全链路约90%
- 任务越长，误差堆积越明显

### 演进时间线
- **2024**: 卷Prompt
- **2025**: 补Context
- **2026**: 讨论慢慢收到Harness

**原因**: 
> 模型智力在线之后，大家开始重新面对软件工程。只不过这次面对的，是一个会推理、会调用工具、还会不断消耗上下文预算的新型系统。

## Harness相关文章脉络

1. **可见性**: OpenAI工程师不写代码了？拆开Harness Engineering看看他们到底在干嘛
   - 仓库知识、UI、日志、指标、CI、PR反馈做成Agent能读取、能执行、能验证的接口

2. **运行时**: 刚刚，Claude Code"代码泄露"背后：如何重新看Agent Harness
   - 主循环、工具池、权限上下文、任务状态和记忆整理

3. **状态治理**: Claude Code长任务为什么不容易跑偏 & 1M上下文不是终点
   - 哪些信息继续留在当前会话，哪些应该回滚，哪些需要压缩，哪些应该清空重开

4. **演进节奏**: Anthropic的Harness，正在进入第二阶段
   - 早期模型扛不住时Harness补结构；模型跨过门槛后Harness要删掉dead weight

5. **信息架构**: 多 Agent不是虚拟公司
   - 上下文边界、信息流和停止条件

6. **软件工程成熟度**: 你的AI-First对了吗？& Google工程师用Claude Code自动化80%？
   - 能不能稳定跑起来，越来越取决于模型外面那层工程系统

## Harness设计取舍

### 多Agent vs 单Agent
**误区**: 一上来就上多Agent

**代价**: 额外路由开销、上下文损失、角色边界设计、更多失败点

**更稳顺序**: 先把单Agent做通，再把明显超载的职责拆出去

### ReAct vs Plan-and-Execute
**ReAct优点**: 灵活，边想边做（像即兴驾驶）
**Plan-and-Execute优点**: 稳定，先画施工图再开工

**选择**: 任务越长、代价越高、回滚越麻烦，越值得把计划层显式拿出来

### Context管理
**误区**: 窗口大一点就好了

**成熟系统关心**: 信号密度
- 哪些信息必须常驻
- 哪些信息只在需要时检索
- 哪些旧Observation应该折叠
- 哪些总结必须进入长期记忆

### 自验证 vs 外部验证
**问题**: 让生成器自己验自己，速度很快，但最容易放过自己

**建议**: 只要任务碰到代码、页面、部署、数据写入这些"能真测"的地方，外部验证几乎都是值得补上的

### Harness厚度
**太薄**: 很多稳定性问题只能靠模型自觉
**太厚**: 系统笨重、昂贵，和当前模型强绑定，模型升级时脚手架成负担

**平衡**: 一边补承重结构，一边等待模型进步后再把不再承重的部分删掉
- Manas案例: 半年内重建五次，每次都在做减法

## 实战建议

### 1. 先让最短链路跑通
用户任务进入后：
- 模型能稳定决定何时调用工具
- 工具结果能正确回写
- 出现失败时不会默默跳过
- 任务结束条件明确

**没有这条主链路，后面加再多记忆、子代理和编排，都不太稳。**

### 2. 工具不是越多越好
- 工具数超过10个且职责重叠时，模型调用质量明显下降
- 更稳做法: 只暴露当前步骤会用到的最小工具集

### 3. 长期记忆要有，但不要越权
- 适合做: 索引、摘要、决策记录
- 关键动作前: 仍要回到真实文件和真实环境做二次确认

### 4. 能写测试就别只写总结
能跑页面就别只看文字描述。
能让lint、类型检查、截图或真实API响应说话，就别只让模型自己评价"应该没问题"。

### 5. 长任务一定会失败
系统设计不该假设它不会失败，而该先假设它必然会在中途被打断、跑偏、超时或遇到工具异常。

有检查点，失败是恢复问题。
没检查点，失败重新变成运气问题。

### 6. 高风险动作必须边界
删除、外发、部署、联网写入、批量修改，这些动作都不该默认和普通读操作一样。

模型可以提出动作。
系统必须决定边界。

## Harness的完整组成

### 不只是Runtime
**误区**: 很多人以为Harness只是Runtime

**真相**: Harness真正吃力的地方，往往是那些把团队经验外移成工件的层。

**核心问题**: 尽量缩小"必须靠模型临场发挥"的面积。

### 三层工件

#### AGENTS.md
**角色**: 仓库地图/默认答案
**回答**: 仓库层面的默认规则是什么

#### Spec
**角色**: 任务契约
**回答**: 本轮任务的上下文管理层

#### Skills
**角色**: 程序性记忆
**回答**: 团队沉淀下来的可复用方法

**合起来做同一件事**: 把团队经验外移成工件，让模型不必临场发挥。

## Harness是AI时代的软件工程接口

**软件工程过去30年反复在做的**: 把复杂系统变成可控系统

**Harness让系统重新变得**: 可设计、可治理、可拆边界

**门槛**: 软件工程能不能被改造成"对Agent可见、可验证、可执行"的系统

**经典引言**:
> 如果你不是模型，你就在做Harness。

这句话把注意力从"怎么把模型哄好"重新拉回到了"怎么把系统搭对"。
