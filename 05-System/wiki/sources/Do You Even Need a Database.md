# Do You Even Need a Database?

**Source URL**: https://www.dbpro.app/blog/do-you-even-need-a-database
**Author**: Jay (DB Pro Team)
**Collected**: 2026-04-19
**Language**: 英文

## 核心论点

大多数早期应用可能不需要数据库。文章通过基准测试对比了三种文件存储策略（线性扫描、内存哈希表、磁盘二分查找）与SQLite的性能，结论是：简单的flat file方案可以支撑数千万日活用户，只有当数据集超过RAM、需要复杂查询、多进程写入或跨实体事务时才真正需要数据库。

## 文章背景

作者来自DB Pro团队（正在构建跨平台的数据库客户端），但文章给出了诚实答案：是否需要数据库取决于规模，而大多数应用的规模比人们想象的小得多。

## 三种文件存储策略

### Approach 1: Linear Scan（每次读取整个文件）
- **实现**: 打开文件，逐行扫描，解析JSON，检查ID
- **复杂度**: O(n)
- **性能**: 10k记录时783 req/s，1M记录时降至23 req/s（Go）

### Approach 2: In-Memory Map（启动时加载到内存）
- **实现**: 启动时读取整个文件，存储在哈希表中，读写都针对map
- **复杂度**: O(1)
- **性能**: 任何规模下97k req/s，亚毫秒延迟（Go）
- **限制**: 需要数据集能放入RAM

### Approach 3: Binary Search on Disk（磁盘二分查找）
- **实现**: 数据按ID排序，构建固定宽度索引（58字节/条），使用ReadAt进行二分查找
- **复杂度**: O(log n)
- **性能**: 10k记录时45k req/s，1M记录时38k req/s（仅下降15%）
- **优势**: 不需要加载全部数据到RAM

## 基准测试结果

### 性能对比（Requests per second，越高越好）

| 方法 | 10k记录 | 100k记录 | 1M记录 |
|------|---------|----------|--------|
| Linear Scan (Go) | 783 | 85 | 23 |
| Binary Search (Go) | 45,742 | 41,661 | 38,866 |
| In-Memory Map (Go) | 97,040 | 98,277 | 97,829 |
| SQLite | 26,000 | 25,507 | 25,085 |
| Binary Search on Disk (Go) | 163,687 | 155,364 | 169,106 |

### 关键发现

1. **线性扫描线性下降**: 1M记录时Go仅23 req/s，Bun平均延迟超过1秒
2. **磁盘二分查找快且平稳**: 数据增长100倍，性能仅下降15%
3. **二分查找击败SQLite**: 在各规模上比SQLite B-tree快1.7倍
4. **SQLite快速且一致**: 无论数据集大小，2.5万req/s，2ms延迟
5. **内存哈希表是天花板**: 任何磁盘方案都无法匹敌
6. **Bun在哈希表方法上击败Go**: 106k vs 97k req/s（运行时优势）
7. **Rust在线性扫描上大幅领先**: 2,883 req/s vs Go的783

## 实际应用规模推算

### 假设条件
- 峰值/平均流量比: 2:1
- 每活跃用户每小时触发10次查询
- 峰值时10%日活用户同时在线
- **公式**: Peak req/s = DAU × 0.10 × (10 lookups/hr ÷ 3600 sec/hr) = **DAU × 0.000278**

### 各方法的承载能力

| 方法 | 峰值容量 | 可支撑日活用户 |
|------|----------|---------------|
| Linear Scan (Go) | 783 req/s | 280万 |
| Binary Search (Go) | 40,000 req/s | 1.44亿 |
| SQLite | 25,000 req/s | 9000万 |
| In-Memory Map (Go) | 97,000 req/s | 3.49亿 |
| Binary Search on Disk (Go) | 169,000 req/s | 6.08亿 |

### 实际案例
- **1万付费客户的SaaS**: 每天使用一次 = ~3 req/s峰值
- **10万日活的消费者应用**: ~30 req/s峰值
- **Instagram**: 4亿日活时仍在用PostgreSQL作为主数据存储

**结论**: 诚实的答案是"你可能还不需要数据库"。当你需要时，单台服务器上的SQLite就能支撑9000万日活用户。

## 什么时候真正需要数据库？

### 1. 数据集不适合RAM
内存哈希表需要启动时加载全部数据。几百万条小记录没问题，但几千万条就需要几GB RAM仅用于索引。

### 2. 需要多字段查询
目前只有"按ID查找"是快速的。如果需要"查找用户X的所有订单"或"查找价格低于50美元的所有产品"，就需要扫描文件或维护额外map。有三四个这样的需求，你就构建了一个查询引擎。

### 3. 需要JOIN
在单个响应中组合orders、users、products意味着从三个文件加载并在应用代码中组装结果。SQL更高效。

### 4. 多进程需要同时写入
单个进程内的RwLock保护并发访问。一旦运行两个服务器实例，各有自己的内存map，它们就会分歧。你需要外部真实来源。

### 5. 需要跨实体原子写入
创建订单同时扣减库存需要要么都成功要么都失败。使用独立文件需要自己实现事务日志。数据库通过ACID保证解决这个问题。

## 技术细节

### 索引格式
固定宽度索引：每条记录一行，精确58字节：`<36-char UUID>:<20-digit byte offset in data file>\n`

固定宽度意味着可以通过单个`ReadAt(buf, entryIndex * 58)`跳转到任何条目。

### 数据文件要求
必须按ID排序后构建索引。在种子时排序一次，或作为一次性迁移步骤。

### 写入策略
追加新记录会破坏排序，实际系统中需要：
- 定期重建索引
- 或保持无序的write-ahead buffer并合并
- 这是LSM-tree所做的合并模式

## 可迁移性

**文件仍在那里，需要时可以迁移**。JSONL可以轻松导入任何数据库。你不会被锁定。
